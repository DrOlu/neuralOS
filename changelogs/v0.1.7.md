# v0.1.7 Release Notes

### English

This release focuses on a smoother chat-first TUI experience, a more practical built-in CLI workflow, and stronger runtime stability.

**What’s New for You**

- **Cleaner TUI layout**: the interface is now more compact and easier to scan during long conversations.
- **Better input experience**:
  - `/` commands and `@` mentions now have clearer autocomplete behavior.
  - `Enter` sends, `Ctrl+J` inserts newline.
  - Multi-line input and clipboard paste behavior are improved.
- **Clearer running feedback**: it is easier to tell when a session is actively running.
- **Better output readability**: assistant/tool messages render in a cleaner and more consistent way.

**Bundled `gyll` CLI (Desktop)**

- GyShell desktop now includes a built-in `gyll` CLI flow.
- If `--url` is not provided, CLI automatically tries the local desktop backend on your machine (default port: `17888`).
- You can use:
  - `gyll --url ip:port`
  - `gyll --url ip:port "Hello"`
  - `gyll run --url ip:port "Do something"`
  - `gyll hook --url ip:port "Send and exit"`
  - `gyll`
  - `gyll "Hello"`
  - `gyll run "Do something"`
  - `gyll hook "Send and exit"`
- Startup behavior:
  - `gyll` (no message): enter interactive TUI.
  - `gyll "message"`: create a new session, send the message immediately, then enter TUI.
  - `gyll run "message"`: create a new session, run without entering TUI, and stream AI output directly in terminal.
  - `gyll hook "message"`: create a new session, send one message, and exit immediately without streaming.
- `--sessionid <id>` is supported to try restoring a specific session on startup.
- CLI endpoint configuration is unified around `--url`.
- Note about `hook--sessionid <id> "msg"`: this mode is designed for callback-style self-wakeup during long workflows. A common pattern is asking the agent to write temporary code that runs a long task, then call `gyll hook ...` at the end (or inside `catch error`) to inject a follow-up/error message back to itself.

**Connection and Settings Improvements**

- WebSocket access control is now more flexible through app-side runtime settings/policy.
- Session lifecycle behavior is more stable, especially around start/stop/ready transitions.
- Settings migration and split are safer and more predictable across updates.

### 中文 (Chinese)

这个版本主要聚焦三件事：更顺手的 chat-first TUI、好用的内置 CLI、以及更稳定的运行体验。

**你能直接感知到的变化**

- **TUI 更清爽**：界面更紧凑，长对话时信息更容易扫读。
- **输入体验更好**：
  - `/` 指令和 `@` 提及的自动补全更清晰。
  - `Enter` 发送，`Ctrl+J` 换行。
  - 多行输入和粘贴体验更稳定。
- **运行状态更清楚**：会话是否在运行中更容易识别。
- **消息显示更干净**：助手与工具输出的展示更一致、更易读。

**桌面版内置 `gyll` CLI**

- GyShell 桌面版现在包含内置 `gyll` CLI 使用链路。
- 如果不传 `--url`，CLI 会自动尝试连接你本机的桌面版后端（默认端口 `17888`）。
- 常见用法：
  - `gyll --url ip:port`
  - `gyll --url ip:port "你好"`
  - `gyll run --url ip:port "执行任务"`
  - `gyll hook --url ip:port "发送后退出"`
  - `gyll`
  - `gyll "你好"`
  - `gyll run "执行任务"`
  - `gyll hook "发送后退出"`
- 启动行为区别：
  - `gyll`（不带消息）：进入交互式 TUI。
  - `gyll "消息"`：新建会话并立即发送消息，然后进入 TUI。
  - `gyll run "消息"`：新建会话，不进入 TUI，直接在终端流式输出 AI 结果。
  - `gyll hook "消息"`：新建会话，只发送一次消息后立即退出，不在终端持续输出。
- 支持 `--sessionid <id>`，启动时优先尝试恢复指定会话。
- CLI 的连接参数统一为 `--url`。
- `hook--sessionid <id> "msg"` 备注：这个模式是为“执行任务时让 Agent 回调自己”设计的。典型用法是让 Agent 写一段临时代码去跑长任务，在代码最后执行一次 `gyll hook ...`，这样程序结束后可以再次唤醒自己；或者在 `catch error` 里调用它，把错误信息回插给自己。

**连接与设置能力改进**

- WebSocket 访问控制通过应用侧策略变得更灵活。
- 会话的开始、停止、完成等生命周期行为更稳定。
- 设置迁移与设置域拆分更安全，版本升级时更可控。
